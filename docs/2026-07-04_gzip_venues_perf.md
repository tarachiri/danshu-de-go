# venues.json 配信サイズ・表示速度調査 【解決済み】

- 記録日: 2026-07-04
- 関連: 断酒でGO (danshu-de-go)
- ステータス: ✅ 解決済み（根本原因特定・対応完了）

## 経緯

venues.json (961KB) の表示が遅い問題を調査。当初、gzip圧縮配信が
未設定なのではと仮説を立て、Cloudflare側の設定確認を試みた。

## 調査内容と詰まった点

- `curl -sI -H "Accept-Encoding: gzip"` でヘッダー確認を試みたが、
  tyo・soi双方から `000 size:0`（DNS/ネットワーク到達不能）。
- tyo/soi自身が自分の公開サイト (danshu-go.nukadokonokai.com) に
  到達できない構成だったため、この方法での確認は断念。
  → 外部ネットワークからの疎通確認、またはブラウザDevToolsでの
    確認に切り替える必要がある（tyo/soiからの自己参照curlは
    今後も使えない前提で調査すること）。

## 判明した根本原因（別セッションで解明済み）

- **gzip圧縮は既にFastly CDN経由で有効になっていた。**
  venues.json は 961KB → **86KB** に圧縮された状態で配信済み。
  → Cloudflare側の設定変更は不要だった（当初の仮説は誤り）。
- 表示が遅い体感の**真因はデータ転送量ではなく、Leafletの
  マーカー描画（レンダリング）処理**だった。

## 対応・現状

- gzip関連：追加対応不要（Fastlyが既に処理済み）。
- 真因（マーカー描画）への対応は、venues.json/app.js/index.html の
  分離作業（generate_map_v5.py導入、index.html 122K→11K等）と
  並行して進行中。マーカー描画自体の最適化（クラスタリング設定見直し等）
  は未着手であれば今後の課題。

## 次に同じ調査をする人（AI含む）へのメモ

- tyo/soiから自サイトへのcurlは通らないネットワーク構成。
  他の経路（外部ホスト、ブラウザDevTools）を使うこと。
- gzip/圧縮の再調査は不要。ボトルネックはフロントのレンダリング側。
