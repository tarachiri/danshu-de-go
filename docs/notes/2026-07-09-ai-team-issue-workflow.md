# 2026-07-09 Claudeかもちゃん・Codexふーちゃん Issue連携メモ

## 目的

Claudeかもちゃん、Codexふーちゃん、まじまじさんが同じチームとして
作業しやすくするため、GitHub Issueとmdファイルを共通ホワイトボードとして使う。

会話がClaude側、Codex側、ローカル作業、gen/tyo本番作業に分かれても、
「何を直すのか」「どこまで確認したのか」「何が未解決か」が散らばらないようにする。

## 役割分担

- Claudeかもちゃん: 構想、設計、論点整理、判断材料、文章化。
- Codexふーちゃん: 現物確認、実装、検証、GitHub Issue/PR/mdへの反映。
- まじまじさん: 方向性判断、実利用での確認、最終承認。
- GitHub Issue: 決定事項、未決事項、次にやること、再発防止チェックリスト。
- mdファイル: 作業記録、調査結果、根本原因、変更点、検証結果。

## Issueの使い方

Issueは一人開発でも複数AI開発でも、共通の作業台として使う。

向いている内容:

- バグ: 例会検索、表示、データ生成、問い合わせ配送など。
- 設計課題: GOGOのDB化、保存API、状態管理など。
- 実装タスク: 小さく切れる作業単位。
- 調査メモ: 原因がまだ確定していない問題。
- 後回し機能: PDF/Excel取り込み、OCR、即時反映など。

書かない方がよい内容:

- APIキー、トークン、LINE UserIDなどの秘密情報。
- 個人情報、問い合わせ本文の生データ。
- 公開リポジトリに出せないサーバー内部情報。

## Issueに残す基本項目

Issue本文またはコメントには、最低限以下を書く。

- 問題: 何が起きているか。
- 影響: 誰に、どの画面・機能で影響するか。
- 原因候補: まだ仮説でもよい。
- 確認済み: 実際に見たファイル、URL、API、ログ。
- 未確認: まだ見ていない経路。
- 完了条件: 何を確認したら閉じてよいか。
- 次の作業: Claude/Codex/まじまじさんの誰が何をするか。

## 複数経路チェックリスト

今回の「今日の例会」バグでは、Web側 `chat.html` は直っていたが、
gen上のFastAPI/LINE断かも `main.py` が未修正で残っていた。

同じ機能を複数経路が持つ場合は、Issueに以下のようなチェックリストを置く。

- 静的Web側を確認したか。
- gen本番API側を確認したか。
- LINE経路を確認したか。
- tyo生成データを確認したか。
- GitHub Pages / 本番URLの反映を確認したか。
- ブラウザキャッシュやJSONキャッシュを考慮したか。
- 作業記録mdを更新したか。

## 断酒でGOで特に注意する経路

- `soi`: 開発作業、GitHub Pages向けフロント編集。
- `tyo`: `danshu.db`、cron、`generate_map_v6.py`、`venues.json`/`schedule.json`生成。
- `gen`: 断かもFastAPI、LINE/WebチャットAPI、AI/RAG系。

一つの問題が複数の機械・複数の実装にまたがることがある。
「直した」と言う前に、どの経路を直したのかを明記する。

## GOGOでのIssue運用

GOGOは `tarachiri/dansyu-gogo` のIssueを主な作業台にする。

親Issueの例:

- GOGOアプリをサーバー側DB保存に移行する設計

小Issueへ分ける候補:

- GOGO用SQLiteスキーマ案。
- 投稿保存APIの最小仕様。
- 管理画面の状態変更UI。
- 確約済みデータの本体連携JSON。
- PDF/Excel/CSV取り込み。
- かもちゃん/断かもの呼称整理。

## 運用ルール

- Claude側で決まった設計は、Issueまたはmdに要点を残す。
- Codex側で実装した変更は、変更点と検証結果をmdに残す。
- 口頭・チャットだけで決めたことを完了扱いにしない。
- 実装前に直近mdを読む。
- 複数AIが同時作業している前提で、未追跡ファイルや既存変更を勝手に戻さない。
- 本番に直接触った場合は、バックアップ名、変更ファイル、再起動方法、検証結果を残す。
