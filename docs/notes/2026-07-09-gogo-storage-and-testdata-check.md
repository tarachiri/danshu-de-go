# 2026-07-09 GOGO保存方式とテストデータ確認

## 今日の確認課題

- 断酒でGOGO試作が `localStorage` 前提になっているか。
- `production-test-data.json` の抽出元が断酒でGO本体のどこか。
- 試作内の「かもちゃん」が、断かもと別人格なのか同一サービス内の呼び方なのか。

## 確認結果

- 現在の試作は `app/app.js` と `app/meetings-table.js` が同じ `localStorage` を読み書きしている。
- そのため、GitHub Pages単体では端末ごとの保存になり、複数端末・複数人共有はできない。
- ただし設計メモでは、一次情報の保管と変換は自宅サーバー側、GitHub Pagesは公開成果物の表示先と整理されている。
- `production-test-data.json` は40件で、断酒でGO本体の `schedule.json` からGOGO用の項目名へ変換したテスト投入用コピーだった。
- 本体の `schedule.json` は tyo の `generate_map_v6.py` が `danshu.db` から生成する公開用JSONなので、DB直エクスポートではなく公開JSON経由の試作用抽出と見るのが正確。
- 「かもちゃん」は、画面上の登録案内役として使われている。既存整理では、断酒でGO本体のWeb/LINEアシスタントは「断かも」と位置づけられているため、将来は呼称を統一するのが安全。

## 変更点

- この確認メモを新規作成した。
- アプリ本体のコード変更は行っていない。

## 次の課題

- GOGOを試作のまま進めるなら `localStorage` 継続でよい。
- 実運用を見据えるなら、先にサーバー側保存APIとSQLiteテーブル設計を決める。
- チャット案内役の表示名を「断かも」に寄せるか、「かもちゃんと登録」を愛称として残すか決める。
