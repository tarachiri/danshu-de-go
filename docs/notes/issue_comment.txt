# 2026-07-10 gogo_organizations と org_hierarchy の名前空間分離・既知の伏線

作成日: 2026-07-10
作成者: かもちゃん（Claude Sonnet）
参照元: `2026-07-10-gogo-schema-review-response.md`（ふーちゃん追記分）
ステータス: 対応不要（今は） / 将来の統合検討事項として記録

## 経緯

ふーちゃんが tyo の `danshu.db` を読み取り専用で確認したところ、GOGO用に
設計していた `organizations` テーブル名が、本体アプリ側の既存テーブル
`org_hierarchy`（`phase1-organizations-design.md` で設計、中身は空）と
目的は違うものの名前が衝突することが判明した。

まじまじさんとふーちゃんの相談により、**GOGO用テーブルは
`gogo_organizations` にリネーム**して名前空間を分離することで決着した。
構造（カラム・CHECK制約・インデックス方針）は変更なし。

## 今回リネームで解決したこと

- `organizations`という同じ名前を、目的の異なる2つのテーブル
  （GOGO権限管理用 / 本体venues・meetings名寄せ用）で使い回す混乱を回避
- 複数AI（かもちゃん・ふーちゃん・チャッピー）が並行作業する環境で、
  名前の一意性による事故防止

## 未解決のまま残る伏線（今すぐの対応は不要）

`gogo_organizations`（GOGO用、責任者・事務局の権限管理が目的）と
`org_hierarchy`（本体用、venues/meetingsの組織名寄せが目的、現在中身は空）
は、**目的は別だが実体としては同じ「断酒会の組織階層」を指す可能性が高い**。

例えば「〇〇県断酒会」という組織は、将来的に

- `gogo_organizations` 側：GOGO投稿の責任者・事務局を管理するために登録
- `org_hierarchy` 側：venues/meetingsの表記ゆれ解消・名寄せのために登録

という形で、**同じ組織名を2つのテーブルに別々に入力することになり得る**。

このまま放置すると起こりうる問題:

- 片方だけ名称変更・組織再編を反映し、もう片方が古いまま残る（データのズレ）
- 「〇〇県断酒会」がGOGO側と本体側で微妙に違う表記になり、突き合わせが困難になる
- 将来、両テーブルを統合したくなったとき、どちらを正とするかの判断が必要になる

## 今回は対応しない理由

- `org_hierarchy` は現時点で中身が空であり、本体側の名寄せ機能自体が
  まだ実装されていないため、今すぐ統合を急ぐ実害がない
- GOGO側の実装（`gogo_organizations`）を先に進めることが優先度として高い
- 統合方針（片方を正とするか、対応表を作るか、将来的に1本化するか）を
  拙速に決めると、GOGO実装のスピードを不必要に落とす可能性がある

## 将来、統合を検討するタイミング

- 本体側の `org_hierarchy` を実際に使う実装（venues/meetingsの名寄せ機能）
  に着手するとき
- `gogo_organizations` にある程度実データが蓄積され、
  「都道府県断酒会」等の組織名の実例が揃ったとき

このどちらかのタイミングで、以下を再検討する。

- 対応表（`gogo_organizations.id` ⇔ `org_hierarchy.id` のマッピングテーブル）
  を作るか
- どちらか一方に統合し、もう一方を廃止するか
- 統合せず、目的別に分離したまま運用を継続するか（データ整合性は運用ルールで担保）
