# インシデント記録：venue_idマッチングの正規表現バグによる誤紐付け

作成: 2026-07-09 (ふーちゃん / Claude Code, soi上)
発端: まじまじさんからの質問「昭島例会がpinで表示されないが、日程で表示されているのはなぜか？」

## 発生した事象

地図上「昭島」を探しても昭島例会のピンが出ない。一方で断かも(chat.html)やDB上には
「昭島例会」という名前の例会自体は存在していた。

調査の結果、meetings.venue_id が実際とは違う会場（東京都世田谷区の「経堂地区会館」
venue_id=9）を指していた。同じ会場に、昭島だけでなく福生・立川・世田谷・中野・墨田・
杉並・荒川・多摩本部など、本来無関係な例会が計17件誤って集約されていた。

## 根本原因

`tama_ical_common.py` の `_normalize_for_match()` にある正規表現

```python
s = re.sub(r'[一-龥]{1,3}[市区]', '', s)
```

は「市区町村名を取り除いて住所の表記揺れを吸収する」ためのものだが、施設名
「経堂**地区**会館」の「地区」（＝○○区）にも誤ってマッチしてしまい、
`"会館"` という極端に汎用的な文字列だけが残る。

`find_venue_id()` は venues テーブルを **id昇順** でスキャンして
「正規化後の文字列が包含関係にあれば一致」と判定するため、"会館" のような
汎用語を含むあらゆるロケーション文字列に対して、id の小さい venue_id=9 が
**最初にヒットしてしまう誤検出**を起こしていた。

同一の正規表現パターンが以下のスクリプトにもコピーされていた：

- `chiba_ical_common.py`
- `kanagawa_ical_common.py`
- `saitama_ical_common.py`

実際に神奈川県（venue_id=90「川崎市中原市民館」→「民館」に潰れる）と
千葉県（venue_id=1796「松戸市民会館」→「民会館」に潰れる）でも、
別の市の例会が誤って紐付いている実害を確認した。

`aichi_match_venues.py` にも類似のパターン（`_CITY_RE`）があるが、
`(?!民)` の除外があり別実装のため、今回は未検証・未修正。

## 対応内容

### スクリプト修正（バックアップ `.bak_20260709` を同ディレクトリに保存）

`tama_ical_common.py` / `kanagawa_ical_common.py` / `chiba_ical_common.py` /
`saitama_ical_common.py` の4本で、正規化後の文字列が **4文字未満なら
マッチに使わない** ガードを追加。テストDBへの複製に対して dry-run で
全国分を検証してから本番に反映した。

```python
if len(fname_norm) >= 4 and (fname_norm in loc_norm or loc_norm in fname_norm):
    return vid
...
if len(addr_norm) >= 4 and (addr_norm in loc_norm or loc_norm in addr_norm):
    return vid
```

### データ修正（本番DBバックアップ `danshu.db.bak_before_*` を保存済み）

- 東京都：昭島例会・立川新年会・連合/多摩総会のvenue_idを実際のiCal
  LOCATIONデータで裏取りして修正。中野・墨田(第1/2)・杉並・荒川(例会/第1/2)・
  多摩家族会/本部例会/酒害相談/連合合同例会/福生例会は、修正後の再同期で
  既に正しいvenue_idを持つ重複行が生成されたため、旧・誤紐付け行を
  `status='inactive'` にして無効化（削除はしていない）。
- 神奈川県：川崎市中原市民館(venue_id=90)に誤って集約されていた
  幸市民館例会・平塚例会・厚木「例会」・多摩市民館例会・宮前市民館例会・
  高津市民館例会・相談·質問(湘南平塚)を修正／無効化。
- 千葉県：松戸市民会館(venue_id=1796)に誤って紐付いていた野田例会を修正。

### フロントエンド反映

tyo上で `generate_map_v6.py` を再実行し `venues.json` を再生成、
`862b59a` としてgit push済み（[danshu-de-go@862b59a](https://github.com/tarachiri/danshu-de-go/commit/862b59a)）。

## 積み残し・今後の課題

1. **`aichi_match_venues.py` の未検証**：同種の市区町村名ストリップ処理があるが、
   実際に誤マッチを起こしているか未確認。
2. **id=104「世田谷例会」、id=163「川崎例会」等の古い放置レコード**：
   day_of_week等のスケジュール情報を持たず、今回の同期でも触られていない。
   実害の証拠がないため現状維持にしているが、いずれ棚卸しが必要。
3. **同じ正規表現の脆弱性がまだ残っている**：4文字未満ガードは対症療法。
   `_normalize_for_match()` 自体が「地区」「市民」等の実在の施設名の
   一部を市区町村名と誤認する構造上の問題は解消していない。将来
   3文字以下の施設名が増えると再発しうる。

## 教訓（今後のスクリプト作成ルールへの反映）

住所・施設名を正規化して**部分文字列の包含判定でマッチさせる**手法は、
正規化後の文字列が短くなるほど誤検出のリスクが跳ね上がる。
`find_venue_id`系の関数を新しい県で複製する際は、正規化結果の
最小文字数ガードを最初から組み込むこと。
