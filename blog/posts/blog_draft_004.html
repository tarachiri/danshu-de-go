なぜraw_meetingsテーブルを作ったのか

今まで例会データを収集するとき、スクリーニング不足と重複リスクという問題があった。各都道府県の公式サイトは構成がバラバラで、命名規則も住所の記載有無もまちまち。そのまま本番のvenues/meetingsに入れてしまうと、誤データが混入したときの影響が大きい。

そこで今日作ったのがraw_meetings（1次情報集積テーブル）だ。

[各ソース] → raw_meetings (status=raw)
                 ↓ 自動スクリーニング
              status = ok / duplicate / suspicious
                 ↓ 人間確認（suspiciousのみ）
              status = ok
                 ↓ promote_to_meetings.py
              venues / meetings（本番）

生データを削除せず常にINSERT ONLYで保持する。原典の証拠として永久保存する設計だ。これで「どのソースから来たデータか」が常に追跡できる。

パイプライン①：haikuちゃんによるHTML抽出

各都道府県の公式サイトはHTMLの構造がバラバラだ。テーブル形式の県もあれば、散文テキストで書かれた県もある。これをパターン別に実装していたら時間がいくらあっても足りない。

そこで登場したのがclaude-haiku-4-5（haikuちゃん）だ。HTMLテキストを渡して「会場名・例会名・曜日・時刻をJSONで返して」と頼むだけで、サイト構成の違いを吸収してくれる。

curl でHTML取得
     ↓
テキスト抽出（script/style除去）
     ↓
haikuちゃんに投げる（max_tokens=8192）
     ↓
JSON配列で返ってくる
     ↓
raw_meetingsにINSERT

パイプライン②：Seleniumによるソーバーねっと全国収集

ソーバーねっと（addiction-peer.net）はReactで動くSPAのため、通常のcurlでは例会データが取れない。JavaScriptを実行できるブラウザが必要だ。

そこで登場したのがヘッドレスChrome + Seleniumだ。画面なしでブラウザをバックグラウンド動作させる腕利きエンジニアの道具だ。

# Ubuntuサーバーでヘッドレスブラウザを動かす
sudo apt-get install -y chromium-browser chromium-chromedriver
pip install selenium --break-system-packages

# ポイント：mtg_result.htmlはURLパラメータで直接検索できる
https://addiction-peer.net/mtg_result.html?pref=43&page=2

このURLパターンを発見したことで、47都道府県×ページネーション対応のスクリプトが書けた。コストはゼロ。夜中にバックグラウンドで全県を回して完了。

瀕死エリアの現実

岩手・宮城・秋田・山形・群馬・石川・鳥取・鹿児島はソーバーねっとにも登録がなかった。デジタル上に情報が存在しない、本当の意味での瀕死エリアだ。

これらの県に対しては、実際に断酒会員と直接会って話を聞くしかない。デジタルで届かないなら人が届く。それが「心の連鎖握手」の本質だと思う。

次のステップ：raw_meetingsからvenuesとmeetingsへの昇格処理。venues一致・meetings未登録の861件をschedule_rawパース（haikuちゃん）→meetings INSERTする。完了すれば現在885件のmeetingsが1700件超になる見込み。

今日学んだこと

「完璧な自動化を最初から目指さない」ということだ。haikuちゃんの出力は確率的で、古いサイトのデータは不正確かもしれない。だからraw_meetingsというクッションを挟んで、人間が確認できる余地を残した。

人命インフラとして、間違った場所への案内は絶対に避けなければならない。精度を上げてから運用の手間を減らし、最終的にコストも下がる。焦らず一つずつ潰していく。それが今日の撃道だった。
